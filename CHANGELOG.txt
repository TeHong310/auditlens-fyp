Date: 2026-06-30
What was changed:
  Backend (auditlens-backend/):
    - config.py rewritten to read all secrets (JWT key, DB credentials,
      Google Vision API key, CORS origins) from environment variables via
      python-dotenv, instead of hardcoding them.
    - Added .env (local only, gitignored) with the previous hardcoded
      values, so local dev behavior is unchanged.
    - requirements.txt trimmed from ~100 packages to the 9 actually used
      (Flask, flask-cors, Flask-JWT-Extended, psycopg2-binary, requests,
      pdf2image, bcrypt, python-dotenv, gunicorn). Removed unused heavy ML
      deps (torch, paddleocr, paddlepaddle, easyocr, opencv, pandas, etc.)
      that were never imported anywhere in the codebase.
    - Added Dockerfile (python:3.11-slim + poppler-utils + gunicorn) so
      PDF->image OCR works on Render's Linux containers (previously relied
      on a hardcoded Windows poppler path).
    - app.py CORS now reads allowed origins from Config.FRONTEND_ORIGINS
      (env var) instead of being hardcoded to localhost:4200.
  Frontend (auditlens-frontend/):
    - Removed unused Angular SSR scaffolding (server.ts, main.server.ts,
      app.config.server.ts, app.routes.server.ts) and related deps
      (@angular/ssr, @angular/platform-server, express, @types/express).
      These were never wired into angular.json's build target, so the app
      was already building as a client-only SPA.
    - Added src/environments/environment.ts (dev) and environment.prod.ts
      (prod), wired into angular.json production build via
      fileReplacements.
    - Replaced hardcoded `private apiUrl = 'http://localhost:5000'` in 9
      components with `environment.apiUrl`, so the API target switches
      automatically between dev and prod builds.
  Deployment:
    - Added render.yaml (repo root) — a Render Blueprint defining:
        auditlens-db (free Postgres), auditlens-backend (Docker web
        service, DB credentials auto-wired from auditlens-db, JWT secret
        auto-generated, Google Vision key set manually), auditlens-frontend
        (free static site with SPA rewrite routing).

Why:
  Migrating AuditLens from localhost-only to a Render deployment. The
  unused ML dependencies and SSR setup would have made the Render build
  fail or require a paid Node service for no benefit; hardcoded secrets
  and localhost URLs needed to become environment-driven so the same code
  works in both local dev and production.

Setup/config needed (manual, on Render dashboard / your machine):
  1. Push to GitHub, then in Render: New -> Blueprint -> connect
     TeHong310/auditlens-fyp -> Render reads render.yaml and creates all
     3 resources. When prompted, set GOOGLE_VISION_API_KEY manually (value
     is in your local auditlens-backend/.env file, never committed).
  2. The new Render Postgres database starts EMPTY — no .sql schema file
     exists in this repo. Before the app will work, export your local
     schema (e.g. `pg_dump -h localhost -U postgres -d auditlens
     --schema-only -f schema.sql`) and load it into Render's database via
     the external connection string shown on the auditlens-db dashboard
     page (`psql "<external-connection-string>" -f schema.sql`).
  3. environment.prod.ts assumes the backend will be reachable at
     https://auditlens-backend.onrender.com (derived from the service name
     in render.yaml). If that name was taken and Render assigned a
     different URL, update environment.prod.ts and redeploy the frontend.
  4. Known limitation (not fixed in this update): uploaded documents are
     saved to a local "uploads/" folder and the file path is stored in the
     DB for later download. Render's filesystem is ephemeral, so uploaded
     files will be lost on every redeploy/restart. Needs a paid persistent
     disk or external object storage (e.g. S3) to fix — flag if you want
     this addressed next.

      GET /ocr-review/invoice/<document_id>/related-docs. For the Finance
      Executive OCR Review page's Invoice tab, returns whether a matching
      PO and GR have been uploaded for a given invoice document, e.g.

  upload-po/<document_id> and upload-gr/<document_id>. Confirmed this
  approach with the user instead of inventing columns that don't exist.
  Frontend (Angular OCR Review component) changes are still pending user
  confirmation that this endpoint works before being built.

Setup/config needed:
  None — no new dependencies, no schema change, no new env vars.

Date: 2026-07-01
What was changed:
  Backend (auditlens-backend/helpers/ocr_helper.py), extract_fields():
    1. Invoice-number "label only, value on next line" pattern required
       neither the no/number keyword nor the colon, so a standalone
       "INVOICE" title line matched it and grabbed the following line
       (e.g. "Bill To:") as the invoice number, then locked out every
       later, more specific invoice-number pattern for the rest of the
       document. Made the no/number keyword and colon mandatory.
    2. Added a pattern for totals written as "TOTAL (RM) 1,150.00"
       (parenthesized currency code breaks all prior amount patterns) to
       both the per-line amount_patterns list and the full-text fallback
       list, inserted first so it's tried before the more specific ones.
    3. Lines like "PO No : PO-2025-0512", "GR No :", "Reg No :" were
       matched by the generic "No\s*:\s*(...)" invoice-number pattern,
       so the PO/GR/registration number got stored as the invoice
       number. Added an early skip for any line containing a
       po/gr/reg/account/tel/fax/bank no. label before running the
       invoice-number patterns on that line.

Why:
  OCR extraction was failing on invoices that use "INVOICE" as a
  standalone title and/or "TOTAL (RM) x.xx" as the total label (e.g.
  INV-ZE-2025-0587), and could misattribute PO/GR reference numbers on
  the same document as the invoice number. Verified with synthetic OCR
  text through extract_fields() directly (dependencies not installed in
  this shell, so tested the function in isolation): the original Zitron
  invoice (INZEPG047575) still extracts unchanged, and the new format
  now correctly extracts invoice_number "INV-ZE-2025-0587" and
  total_amount 1150.00, including when PO/GR/Reg No lines appear before
  the invoice number label in the document.

  Not fixed (out of scope, flagged for awareness): a pre-existing,
  unrelated bug in the same function strips a trailing "-<digits>" from
  invoice numbers matched on a single line (e.g. "Invoice No:
  INV-ZE-2025-0587" on one line truncates to "INV-ZE-2025"). It only
  showed up while writing a stricter test case and isn't part of what
  was reported; needs a separate look if same-line (not label-then-
  next-line) hyphenated invoice numbers turn out to be common in real
  scans.

Setup/config needed:
  None — no new dependencies, no schema/env changes. Requires the Render
  backend service to redeploy so OCR extraction picks up these fixes.

Date: 2026-07-01
What was changed:
  Backend (auditlens-backend/):
    - config.py: added Config.GEMINI_API_KEY = os.environ.get('GEMINI_API_KEY').
    - Added helpers/gemini_extractor.py: calls the Gemini 2.0 Flash REST
      endpoint (generativelanguage.googleapis.com, no new SDK dependency,
      uses the existing `requests` package) with three functions —
      gemini_extract_invoice/po/gr(ocr_text) — each sending a strict
      JSON-only prompt (temperature 0, responseMimeType
      application/json) with document-specific vendor-detection rules
      (e.g. on a PO the header company is the buyer, not the vendor).
      Returns {} on missing key or any error; never raises.
    - helpers/ocr_helper.py: extract_fields(), extract_po_fields(), and
      extract_gr_fields() each gained a fallback block right before
      their final return — if any field is still None after regex
      (currency excluded for PO/GR, since it defaults to 'MYR'), call
      the matching gemini_extract_* function and fill only the fields
      regex left as None. If regex already filled every field, Gemini
      is never called. No existing regex pattern, run_ocr(), or
      run_google_vision_ocr() was touched.

Why:
  Regex extraction works for known layouts but misses on new ones and
  can't apply semantic rules (e.g. distinguishing the issuing vendor
  from the buyer named in the document header, which differs between
  invoices, POs, and GRs). Gemini is added purely as a fallback for
  fields regex leaves blank, keeping the existing regex path as primary
  and free, so LLM cost is only incurred on genuine misses.

  Verified with mocked Gemini responses (real dependencies + the actual
  local GEMINI_API_KEY were used for a first live call, then mocked for
  the rest — see security note below): the original Zitron invoice
  still extracts every field from regex alone with Gemini never called;
  a synthetic invoice with no regex-recognizable labels at all correctly
  triggers the fallback and gets invoice_number/total_amount filled from
  the (mocked) Gemini response; the same missing-field fallback
  mechanism was confirmed for PO and GR extraction.

  IMPORTANT — known gap, not addressed here: the fallback only fills
  fields regex left as None. It does NOT re-check or correct fields
  where regex already produced a value, even if that value is wrong
  (e.g. regex picking the buyer's name as vendor_name, or stripping a
  "PO-" prefix off a po_number). This matches the exact merge logic
  specified for this change, but means a real PO/GR where regex returns
  a wrong-but-non-null vendor/number won't get corrected by Gemini as
  currently wired. Flagged for a decision on whether wrong non-null
  values should also be eligible for Gemini correction.

  SECURITY NOTE: while testing live, a rate-limit (429) error from
  `requests` embedded the full request URL (including the ?key=...
  query parameter) in its exception text, which got printed via a
  DEBUG log during testing and exposed the real GEMINI_API_KEY in the
  session transcript. Fixed by sending the key via the `x-goog-api-key`
  HTTP header instead of a URL query parameter (still no new
  dependency), so it can no longer leak through error messages, proxy
  logs, or redirects. The exposed key should be rotated regardless.

Setup/config needed:
  1. GEMINI_API_KEY must be set in both the local .env file and Render's
     environment variables for auditlens-backend (confirm it's present
     in both places — the code returns {} and falls back to regex-only
     behavior if it's missing, but the semantic fallback won't run).
  2. Rotate/regenerate the GEMINI_API_KEY that was exposed during this
     session's live test (see security note above) before relying on
     the old value in production.
  3. Redeploy the Render backend service after pushing.

Date: 2026-07-01
What was changed:
  Backend (auditlens-backend/helpers/ocr_helper.py):
    - extract_po_fields() and extract_gr_fields(): replaced the
      "Gemini fills only if regex left the field None" fallback block
      (added in the previous entry) with a "Gemini primary, regex
      secondary" strategy. Gemini is now always called for PO/GR, and
      its result overrides regex for po_number/vendor_name/po_date/
      total_amount (PO) and gr_number/vendor_name/receipt_date (GR)
      whenever Gemini returns a non-null value for that field. Regex
      values are kept only if Gemini returns {} or raises.
    - extract_fields() (invoice) is UNCHANGED: still regex-primary,
      Gemini only fills fields regex left None.
    - helpers/gemini_extractor.py untouched — its PO/GR prompts already
      encode the buyer-vs-vendor distinction, which is exactly why they
      can be trusted as primary for these two document types.

Why:
  Real-world testing on PO-2025-0512.pdf and GR-2025-0520.pdf showed
  regex reliably fills PO/GR fields with a WRONG but non-null value
  rather than leaving them None, because PO/GR flip the semantic role
  of the header company (header = buyer, not vendor) in a way regex
  pattern-matching cannot reason about. The previous "fill only if
  None" rule (correctly implemented per the prior request, but flagged
  at the time as a known gap) never had a chance to correct these,
  since the fields were never actually None. Concretely: regex returned
  po_number "2025-0512" (stripped the "PO-" prefix from a bare
  "PO-2025-0512" line) and vendor_name "KDU TECH SDN. BHD" (the buyer,
  read from the document header) instead of the actual vendor
  "ZITRON ENTERPRISE SDN. BHD."; the same buyer-as-vendor and
  prefix-stripping pattern showed up on the GR. Invoice extraction
  keeps regex as primary because there the header genuinely is the
  seller/vendor, which regex already handles reliably (confirmed
  unchanged: Zitron invoice still fully regex-extracted with Gemini
  never called; INV-ZE-2025-0587 still correctly filled via the
  regex-primary/Gemini-fallback path).

  Verified with mocked Gemini responses reproducing the reported
  failure: reconstructed OCR text that makes regex alone return
  po_number "2025-0512" / vendor_name "KDU TECH SDN. BHD" (PO) and
  gr_number "2025-0520" / vendor_name "KDU TECH SDN. BHD" (GR) —
  confirming the bug is real and not fixed by the previous None-only
  fallback — then confirmed Gemini-as-primary correctly overrides all
  of them to the expected values. Also verified regex values are kept
  as a safety net when the Gemini call raises an exception.

Cost impact:
  Gemini is now called on every PO/GR upload (not just on a regex
  miss), roughly doubling call volume for those two document types.
  At Gemini 2.0 Flash's free tier (1500 requests/day), this stays well
  within limits for demo and typical SME upload volume.

Setup/config needed:
  None beyond the previous entry (GEMINI_API_KEY still required in both
  local .env and Render). Redeploy the Render backend service after
  pushing.

Date: 2026-07-02
What was changed:
  Frontend (auditlens-frontend/src/app/finance/ocr-review/):
    - finance-ocr-review.component.html:
      * GR tab list table: replaced the "Amount" column with "Date"
        (bound to formatDate(gr.receipt_date)) — columns are now File
        Name, GR Number, Vendor, Date.
      * GR detail panel: removed the "Total Amount (RM)" field entirely.
      * Invoice detail panel: "Tax Amount (RM)" field is now wrapped in
        *ngIf="showTaxAmount" (hidden, no "Not detected" placeholder,
        when the invoice has no tax). Added a "+ Add Tax Amount" button
        shown when the field is hidden, which reveals it for manual
        entry. Total Amount is unchanged (still always shown, still
        shows "Not detected" when missing).
    - finance-ocr-review.component.ts: added showTaxAmount boolean and
      addTaxAmount() method; selectDocument() now sets showTaxAmount
      based on whether the selected invoice's tax_amount is non-null.
    - finance-ocr-review.component.css: added .btn-add-tax pill-button
      style (light blue, dashed border, matches existing theme).
    - editGRFields.total_amount (component state) and the GR save
      request payload were deliberately left unchanged/still populated
      from the loaded GR record, even though the input is gone from the
      UI — see rationale below.
  Backend (auditlens-backend/helpers/ocr_helper.py), extract_fields():
    - Fixed invoice-number truncation: the post-match cleanup
      `val = re.sub(r'\s*-\s*\d+$', '', val).strip()` stripped ANY
      trailing "-<digits>", including one that's part of the actual
      invoice number (e.g. "INV-SE-2025-1001" -> "INV-SE-2025"). Changed
      to `re.sub(r'\s+-\s+\d+$', '', val)`, which only strips a trailing
      "- N" when there's whitespace on both sides of the hyphen (a
      page-number-style suffix like "INV12345 - 3"), leaving
      whitespace-free hyphens that are part of the number itself alone.

Why:
  UI change: GR documents don't carry a monetary total (that's on
  Invoices/POs), so showing "Total Amount (RM): Not detected" on every
  GR made users think OCR had failed when the field simply doesn't
  exist on that document type. Many Malaysian SME invoices are
  SST-exempt or from non-registered vendors and have no tax line, so
  the same "Not detected" placeholder was misleading there too — hiding
  it when null (with a manual "+ Add Tax Amount" escape hatch) removes
  the false-failure signal while keeping the field available when
  actually needed. Total Amount keeps "Not detected" since every
  invoice is expected to have one, so a missing value there is a real
  OCR miss worth flagging.

  IMPLEMENTATION NOTE on the GR total_amount removal: the PUT
  /documents/gr/<id>/update backend route (documents.py) unconditionally
  writes `total_amount = data.get('total_amount') or None` on every
  save. If the frontend had stopped sending total_amount in the payload
  entirely, every "Save GR Changes" click would have silently wiped any
  total_amount already stored from OCR/Gemini extraction — even though
  the task explicitly said the field should stay in the model
  ("optional/nullable for backward compatibility") rather than be
  destroyed. So editGRFields.total_amount and the save payload were
  left as-is; only the visible input and its label were removed. Net
  effect: no editable/visible Total Amount on GR, but no silent data
  loss on save. Flagging this judgment call in case a stricter
  interpretation (also strip the field from the save payload) was
  intended instead.

  Bug fix: reported on invoice "INV-SE-2025-1001.pdf" — regex correctly
  captured the full "INV-SE-2025-1001" via
  `([A-Za-z0-9\-\/]+)`, but a subsequent cleanup line meant to strip
  page-number artifacts like "INV12345 - 3" used a regex with optional
  (`\s*`) whitespace around the hyphen, so it also matched and removed
  the legitimate "-1001" segment. Requiring at least one space on each
  side of the hyphen distinguishes an artifact suffix from a hyphen
  that's actually part of the invoice number.

  Verified (mocked Gemini, dependencies stubbed as in prior entries):
  "INV-SE-2025-1001" style text now extracts invoice_number
  "INV-SE-2025-1001" in full; the original Zitron invoice
  (INZEPG047575, no hyphens) is unaffected; a synthetic
  "INV12345 - 3" (space-hyphen-space-digit suffix) still correctly
  strips down to "INV12345".

Setup/config needed:
  None — no new dependencies, no schema/env changes.
  1. Redeploy the Render frontend (static site) to pick up the GR/Tax
     UI changes.
  2. Redeploy the Render backend to pick up the invoice-number regex
     fix.
  3. Delete the existing truncated "INV-SE-2025-1001" document (it has
     invoice_number "INV-SE-2025" stored) and re-upload the same PDF to
     confirm it now extracts the full number.

Date: 2026-07-02
What was changed:
  Backend:
    - auditlens-backend/routes/documents.py: fixed the pre-existing
      GET /documents/<document_id>/file (invoice) route — it hardcoded
      mimetype='application/pdf' regardless of actual file type and had
      no access check. Now guesses the correct Content-Type from the
      file extension (pdf/jpg/jpeg/png are all uploadable types) and
      returns 403 if a finance_executive requests a document they didn't
      upload (auditors are unrestricted, matching the rest of the app's
      access pattern). Added two new sibling routes, since POs and GRs
      are stored in their own tables with their own primary keys
      (po_id/gr_id), not as rows in `documents`:
        GET /documents/po/<po_id>/file
        GET /documents/gr/<gr_id>/file
      All three share a new _send_document_file() helper.
    - Added routes/auditor.py (new auditor_bp, registered at url_prefix
      '/auditor' in app.py):
        GET /auditor/record/<invoice_document_id>/comparison
      Auditor-only. Returns the invoice + its linked PO/GR (via the
      existing document_id foreign key — same linkage used everywhere
      else in this app) plus a computed match_result: vendor_match
      (fuzzy: case/punctuation-insensitive, strips "Sdn Bhd"/"Berhad"),
      amount_match (Invoice vs PO only — GR carries no monetary total),
      date_order_valid (PO date <= GR date <= Invoice date, skipped if
      any date or document is missing), and overall_status computed as
      FAIL if any applicable check fails, PARTIAL if PO or GR is
      missing (but what's present is consistent), else PASS. Never
      raises on missing PO/GR/dates — every check degrades to
      "not applicable" rather than crashing.
  Frontend (auditlens-frontend/src/app/auditor/record-detail/):
    - Full redesign of the Auditor Record Detail page around the new
      comparison endpoint, replacing the old "Run Matching" /
      three_way_matches-based flow entirely:
      * Overall status banner (PASS green / PARTIAL yellow / FAIL red)
        at the top with upload date + OCR confidence.
      * Field-by-field comparison table (Vendor, Amount, PO Ref, Date,
        Doc No rows x Invoice/PO/GR columns) in a monospace font, with
        inline =/≠ pill symbols computed client-side from the raw field
        values already in the payload (so the table can show a
        per-column relationship, not just one aggregate flag per row).
        Mismatch rows get a light-red background. A missing PO or GR
        renders its whole column as a greyed "Not Uploaded" cell instead
        of blank/crashing.
      * Source Documents section with View Invoice/PO/GR buttons, each
        disabled with a "Not uploaded" tooltip when that document
        doesn't exist.
      * A PDF quick-view modal (inline in the same component — no
        separate modal component file, since it's single-use and tied
        to this page's state): fetches the file via HttpClient with the

        used by the Finance OCR Review page's viewDocument()), turns it
        into a blob URL via DomSanitizer.bypassSecurityTrustResourceUrl
        for the iframe, with a loading spinner, Open in New Tab /
        Download / Close footer actions, and closes on Escape or a
        backdrop click. No JWT is ever put in a URL.
      * Kept: Remarks/Notes textarea, Approve / Need Review / Send Back

        comparison endpoint): the old "Run Matching" button, match-score

    - Rewrote the component's .css; most of the old classes (info-bar,
      match-score-badge, docs-status, exceptions-*, run-match button)
      became unused once the layout changed, so the file was replaced
      rather than patched around dead selectors.

Why:
